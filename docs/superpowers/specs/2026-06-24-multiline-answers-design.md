# 멀티라인 답변 파서 — 설계 문서

> 작성일: 2026-06-24
> 상태: 승인됨
> 관련 ROADMAP: 노트→카드 파이프라인 강화 (5-4 이미지 카드의 선행/대체 — 파서 기반 강화)

## 배경 / 목적

앱은 답변(answer)에 코드블록·여러 문단·리스트가 담긴 카드를 렌더링하도록 설계됐다(react-markdown + rehype-highlight). 데모 `public/cards.json` 15장 중 13장이 멀티라인, 3장은 ```코드펜스```를 담는다 — 이것이 "개발자 면접 카드" 정체성의 핵심이다.

그러나 실제 파서 `src/lib/obsidian.ts`의 `parseVault`는 **라인 기반**으로, `질문?::답변`의 `::` 뒤 **한 줄만** 답변으로 가져온다. 즉 데모 카드는 손으로 작성된 것이고, 실제 Obsidian 노트를 파서로 돌리면 그 풍부한 답변을 **만들 수 없다.** 이 작업은 파서가 멀티라인 답변(코드펜스·문단·리스트)을 노트에서 추출하도록 강화한다.

파서는 **in-app 가져오기 · CLI(5-2) · Obsidian 플러그인(5-3)** 세 진입점이 공유하므로, 이 한 곳을 고치면 셋 다 동시에 강화된다.

## 채택 접근: `::` 이어쓰기 (펜스 인식, 하위호환)

기존 `질문?::답변` 문법을 유지하되, `::` 뒤 답변이 **다음 카드 시작/경계까지 여러 줄로 이어지게** 한다. 코드펜스/인라인코드를 인식해 그 안의 `::`를 새 카드로 오인하지 않는다.

### 비목표 (YAGNI)
- 이미지 임베드 (별도 작업 5-4)
- Card/cards.json 스키마 변경
- 중첩 코드펜스 지원
- `?` 단독줄 구분자 등 대체 문법

## 범위 / 표면적

**변경 파일: `src/lib/obsidian.ts`의 `parseVault` 내부 루프만.** 라인별 즉시 방출 → 상태 기반 블록 누적으로. `VaultFile`·`slugify`·`hashContent`·id 규칙·덱 집계·반환 타입(`CardsData`)은 불변. 다른 모듈(serialize/CLI/플러그인/merge) 변경 없음.

## 파싱 알고리즘

스캔의 전처리는 동일하다:
- `## Self-Test Anchors` 이하만 스캔(`inSection`).
- `#flashcard/…` → 현재 `deck`.
- `### Foundation|Mechanism|Diagnosis`(대소문자 무관) → 현재 `tier`.

카드 누적(신규):
1. **카드 시작**: `deck`·`tier`가 설정된 상태에서 "카드-시작 줄"(아래 판별)을 만나면 직전에 열려 있던 카드를 방출(flush)하고 새 카드를 연다. `question` = 원본 줄의 첫 `::` 앞(trim), `answer` 누적 버퍼 = 첫 `::` 뒤(개행 없이 그대로).
2. **이어쓰기**: 카드가 열려 있는 동안, 이후 줄을 `answer` 버퍼에 **개행(`\n`)과 함께 그대로** 덧붙인다. 단 아래 종결자를 만나면 현재 카드를 flush하고 누적을 멈춘다.
3. **종결자(현재 카드를 닫는 것)**: ① 다음 카드-시작 줄, ② `### 티어` 헤딩, ③ `#flashcard/…` 덱 태그, ④ 임의의 ATX 헤딩 줄(`^#{1,6}\s` — 섹션 경계), ⑤ 파일 끝(루프 종료 후 flush). (②③은 동시에 deck/tier 상태도 갱신.)
4. **flush 시 마무리**: `answer`의 **앞뒤 빈 줄/공백만 trim**(`replace(/^\s+|\s+$/g)` 또는 동등). 내부 개행·코드펜스·리스트·빈 줄은 **보존**(react-markdown 렌더 입력). `question`이 빈 카드는 방출하지 않음(기존 동작 유지). `answer`는 빈 문자열 허용.
5. id/sourceHash: 기존과 동일 — `id = {slug}-{tierAbbr}{seq}`(티어별 순번), `sourceHash = hashContent(\`${question}\0${answer}\`)`.

### 카드-시작 판별 (오인 방지 — 핵심)

한 줄이 "카드-시작"인지 판정할 때 **코드를 먼저 가린 뒤** raw `::` 존재를 본다:

- **펜스 블록 상태**: 줄이 ` ``` ` 또는 `~~~`(3개 이상)로 시작하면 `inFence`를 토글한다. `inFence`인 동안의 모든 줄은 **answer 본문**으로만 취급 — 카드-시작/종결자 판정을 하지 않는다(열린 카드가 있으면 그대로 버퍼에 추가). 펜스 여는 줄/닫는 줄 자체도 answer 본문에 포함된다. 중첩 펜스는 미지원(단순 토글).
- **인라인 코드 제거**: `inFence`가 아닌 줄에서, 카드-시작 검사 *전에만* 백틱 인라인 스팬(`` `...` ``)을 제거한 사본을 만든다. 그 사본에 `::`가 남고 그 앞에 비공백 텍스트가 있으면 → **카드-시작**.
- 실제 question/answer 분리는 **원본 줄**의 첫 `::` 기준(인라인 제거본은 판정용일 뿐).
- 즉 `` `std::vector` `` ·`` `key::value` `` 처럼 `::`가 인라인 코드 안이면 카드-시작이 아니다.

**남는 제약(문서화):** 코드펜스/인라인코드 밖 *평문* 줄에 날것 `x::y`가 있으면 새 카드로 오인된다. 개발자 노트에서 `::`는 거의 항상 코드이므로 백틱/펜스로 감싸면 안전하다. ATX 헤딩(`#`)으로 시작하는 줄은 (인라인 제거 후) `::`가 있어도 카드-시작이 아니라 종결자(헤딩)로 우선 처리한다.

### 처리 우선순위 (한 줄당)
1. `inFence`이면 → (펜스 토글 검사 후) answer 본문에 추가, 계속.
2. 펜스 토글 줄(` ``` `/`~~~`)이면 → `inFence` 토글, answer 본문에 추가(카드 열려 있을 때), 계속.
3. `### tier` → 현재 카드 flush, tier 갱신.
4. `#flashcard/…` → 현재 카드 flush, deck 갱신.
5. 기타 ATX 헤딩(`^#{1,6}\s`) → 현재 카드 flush(섹션 경계). (anchor 위쪽은 이미 `inSection`으로 차단됨.)
6. 카드-시작 줄(위 판별) → 현재 카드 flush, 새 카드 open.
7. 그 외(카드 열림) → answer 본문에 추가. (카드 안 열림 → 무시.)

## 하위호환

- 기존 단일 줄 `질문?::답변`은 그대로 동작: 카드 open 후 다음 줄이 보통 카드-시작/헤딩/EOF라 즉시 flush되어 답변은 그 줄의 `::` 뒤 텍스트.
- 카드가 열리기 *전*(티어/덱만 있고 첫 카드 전)의 비-카드 줄은 어떤 answer에도 들어가지 않고 무시됨 → 기존 "메모 줄 무시" 동작 유지.
- **기존 `obsidian.test.ts` 10개 케이스 전부 그대로 통과해야 한다**(회귀 가드). 특히 "splits only on the first ::"(단일 줄, 답변 내 `::`)와 "skips lines without ::"(카드 전 메모 줄).

## 엣지 케이스 (테스트로 고정)

| 케이스 | 기대 |
|--------|------|
| 단일 줄 카드 | 기존과 동일 |
| `질문?::` + 다음 줄들 코드펜스 | 답변에 코드펜스 포함(개행 보존) |
| 펜스 안 `key::value` | 새 카드 아님, 답변에 포함 |
| 인라인 `` `std::x` `` 가 있는 이어쓰기 줄 | 새 카드 아님 |
| 여러 문단(빈 줄 구분) 답변 | 내부 빈 줄 보존, 앞뒤만 trim |
| 연속 단일 줄 카드 2개 | 각각 분리, 답변 안 섞임 |
| 파일 끝에서 끝나는 멀티라인 답변 | EOF에서 flush |
| 답변 도중 `### Mechanism` | 답변 종료 + 티어 전환 |
| 코드펜스/인라인 밖 평문 `x::y` 이어쓰기 줄 | (문서화된 제약) 새 카드로 오인됨 — 테스트로 동작 명시 |
